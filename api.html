Authentication & Security

Accessing Elvion AI's protected endpoints requires a valid API Key. This key must be securely transmitted as a Bearer Token within the Authorization header of your HTTP requests. It acts as your digital key, granting access to the intelligent realms of our platform.

Authorization: Bearer YOUR_ELVION_API_KEY

API Key Best Practices

  • Keep it Secret: Your Elvion AI API Key is a sensitive credential. Treat it with the same care you would a password. Never commit it directly to your source code repositories.
  • Environment Variables: The most secure way to manage your API key is to store it as an environment variable on your server or in a secure secrets management service. Your application can then access it dynamically.
  • Avoid Client-Side Exposure: Do not embed your API key directly in client-side code (e.g., JavaScript in a web browser, mobile app bundles). If client-side access is necessary, route requests through your own backend, which then adds the API key.
  • Rotation: Regularly rotate your API keys. If a key is compromised, you should be able to revoke it and generate a new one. (Feature for future implementation on Elvion AI portal).

Each successful invocation of a protected endpoint consumes credits from your account. An insufficient credit balance will result in a 402 Payment Required HTTP status, prompting you to replenish your digital coffers on the Elvion AI Developer Portal.

CRITICAL: Understand Your API Keys!

It is paramount to grasp the fundamental distinction between two types of API keys relevant to your Elvion AI integration:

  • Your Developers' API Key (YOUR_ELVION_API_KEY): This is the key you provide to your end-users or client-side developers. It authenticates requests originating from your users to your custom Elvion AI backend. Your backend is responsible for validating this key, deducting credits from the associated user account, and then forwarding the request (using its *own* internal keys) to the external AI services. This key should be safeguarded by your users.
  • Your Backend's Internal API Keys (e.g., THENA_API_KEY, PPLX_API_KEY, MINIMAX_API_KEY): These are environment variables that your Elvion AI backend itself uses to authenticate with the external third-party AI services (such as Thena for image generation, Perplexity for advanced chat, MiniMax for other models) that it integrates. These keys are server-side secrets and should never be exposed to your end-users or included in client-side code.

The common errors you might encounter (e.g., "Thena API key not configured") explicitly indicate that your backend deployment (e.g., on Railway.app) is missing these *internal environment variables***. You are required to configure these on your hosting platform's environment settings, not to pass them in the user's API Key. The internal keys are secrets used solely by your backend; your end-users or developers consuming *your* Elvion AI API should never see or provide them.

Action Required: Immediately ensure that environment variables like THENA_API_KEY, PPLX_API_KEY, and MINIMAX_API_KEY (and any others your backend relies on) are correctly set and available to your backend application on your deployment platform (e.g., Railway.app Environment Variables).

Error Handling & Troubleshooting

While interacting with the Elvion AI API, you may encounter various responses indicating either success or a problem. Robust error handling is crucial for building reliable applications. This section outlines common error structures and provides guidance on troubleshooting.

Standard Error Response Structure

When an error occurs, the API will typically return a JSON object with the following structure, providing clear details about the issue:

{
  "detail": "Descriptive error message.",
  "status_code": 400,
  "error_type": "ValidationError"
}
  • detail (string): A human-readable description of the error.
  • status_code (integer): The HTTP status code indicating the type of error.
  • error_type (string): A more specific classification of the error (e.g., "AuthenticationError", "PaymentRequired", "ConfigurationError").

Common HTTP Status Codes

Familiarize yourself with these standard HTTP status codes to quickly diagnose issues:

Status CodeMeaningDescription
200 OKSuccessThe request was successful, and the response body contains the requested data.
400 Bad RequestClient ErrorThe request could not be understood or processed due to invalid syntax or parameters. Check your request body and headers.
401 UnauthorizedAuthentication FailureThe request lacks valid authentication credentials. Ensure your Authorization header contains a correct and active Elvion AI API Key.
402 Payment RequiredInsufficient CreditsYour Elvion AI account has insufficient credits to fulfill the request. Please top up your balance.
403 ForbiddenAccess DeniedYou do not have the necessary permissions to access this resource, even with valid authentication. This might indicate API key scope issues or restricted features.
404 Not FoundEndpoint MissingThe requested API endpoint does not exist. Double-check the URL path.
429 Too Many RequestsRate Limit ExceededYou have sent too many requests in a given amount of time. Implement back-off strategies and respect rate limits. See Rate Limiting.
500 Internal Server ErrorServer-Side IssueAn unexpected error occurred on the Elvion AI backend. This often indicates a problem with internal service configuration (like missing external AI service keys) or an unhandled exception. Refer to the Critical API Keys section if this persists.
503 Service UnavailableTemporary OverloadThe server is currently unable to handle the request due to temporary overload or maintenance. It typically resolves itself; retry after a short delay.

Rate Limiting

To ensure fair usage, maintain service stability, and protect against abuse, Elvion AI enforces rate limits on API requests. If you exceed these limits, you will receive a 429 Too Many Requests HTTP status code.

Understanding Rate Limits

Specific rate limits vary by endpoint and your subscription tier. While precise numbers are not currently public, a general approach is advised:

  • Concurrent Requests: There may be a limit on the number of simultaneous requests you can make.
  • Requests Per Time Unit: Limits are typically applied per minute or per hour for each API key.

Strategies for Handling Rate Limits

To gracefully handle 429 responses and prevent service interruptions, implement the following strategies:

  1. Exponential Backoff: When a 429 is received, don't immediately retry the request. Instead, wait for a short period (e.g., 1 second) and then retry. If it fails again, double the wait time (2 seconds, then 4, 8, etc.). Introduce a maximum number of retries and a maximum wait time.
  2. Jitter: To avoid thundering herd problems (many clients retrying at the same time), add a small random delay (jitter) to your backoff intervals.
  3. Queueing: For high-volume applications, consider implementing a local queue for outgoing API requests. This allows you to control the rate at which requests are sent to Elvion AI.
  4. Credit Monitoring: Actively monitor your Elvion AI credit balance through the developer portal to proactively address potential 402 Payment Required errors before they halt your application.

If you anticipate very high usage requirements that exceed typical rate limits, please contact Elvion AI support to discuss custom solutions or higher tiers.

  "history": [

  "stream": false

Successful Response Structure (stream: false)

When stream is set to false, you will receive a single JSON object:

{
  "response": "Deploying autonomous AI in healthcare raises several key ethical considerations. Foremost among these is...",
  "model_name": "Seraphina v1.0",
  "generated_at": "2026-01-09T14:30:00Z",
  "usage": {
    "prompt_tokens": 50,
    "completion_tokens": 200,
    "total_tokens": 250
  }
}
  • response (string): Seraphina's complete generated answer to the prompt.
  • model_name (string): The identifier for the Seraphina model version used.
  • generated_at (string): Timestamp of when the response was generated (ISO 8601 format).
  • usage (object): Provides details on token consumption, useful for monitoring costs.

Successful Response Structure (stream: true)

When stream is set to true, the API will send a series of Server-Sent Events (SSE), each containing a small piece of the response. The format will typically be:

data: {"token": "Deploying", "event": "chunk"}

data: {"token": " autonomous", "event": "chunk"}

data: {"token": " AI", "event": "chunk"}

...

data: {"response": "Deploying autonomous AI...", "event": "final"}

Your client-side implementation should listen for these events and append the token values to construct the full response. The stream typically concludes with a final event containing the complete response.

-H "Authorization: Bearer YOUR_ELVION_API_KEY" \
-H "Content-Type: application/json" \
-d '{
  "message": "What are the latest advancements in AI-driven drug discovery?",
  "stream": false,

async function generateElvionImage() {
    try {
        const response = await fetch(API_URL, {
            method: 'POST',
            headers: {
                'Authorization': `Bearer ${ELVION_API_KEY}`,
                'Content-Type': 'application/json'
            },
            body: JSON.stringify(data)

        if (!response.ok) {
            const err = await response.json();
            throw new Error(err.detail || `HTTP error! status: ${response.status}`);
        }
        const result = await response.json();
        if (result.image_b64) {
            const imageUrl = `data:image/png;base64,${result.image_b64}`;
            console.log("Image received. You can use the base64 data to display it.");
            // Example: document.getElementById('generated-image-display').src = imageUrl;
            // and document.getElementById('generated-image-display').style.display = 'block';
        } else {
            console.log("No image data received in response.");
        }
    } catch (error) {
        console.error("Failed to generate image:", error.message);
    }
}

Integration Best Practices

To maximize the performance, reliability, and cost-efficiency of your Elvion AI integration, consider these best practices:

  • Error Handling: Always implement robust error handling for all API calls. Anticipate network issues, authentication failures, rate limit breaches, and server-side errors. Log errors comprehensively to aid in debugging.
  • Asynchronous Operations: AI model inference can be time-consuming. Use asynchronous programming patterns (e.g., promises in JavaScript, async/await in Python) to avoid blocking your application's main thread and ensure a responsive user experience.
  • Input Validation: Before sending data to Elvion AI, validate all user inputs on your side. This prevents sending malformed requests and potentially reduces unnecessary credit consumption.
  • Prompt Engineering: The quality of your prompts directly impacts the quality of AI responses. Experiment with different phrasing, add context, specify desired output formats, and iterate to find what works best for each model and use case.
  • Cost Management: Monitor your credit usage regularly through the Elvion AI developer portal. For production applications, consider implementing a budget alert system on your end.
  • Security: Protect your API keys rigorously. Never hardcode them, and restrict access to them. Use environment variables or secure secret management services.
  • User Feedback: If integrating AI into a user-facing application, gather user feedback on AI responses. This data can be invaluable for refining your prompts and improving the overall user experience.
  • Client-Side Streaming (for Chat): For chat applications, enable `stream: true` where available. This provides a faster, more interactive experience for users as they see the response being generated in real-time.

Glossary of Terms

A quick reference for key terminology used throughout the Elvion AI API documentation.

API (Application Programming Interface)
A set of rules and protocols for building and interacting with software applications. The Elvion AI API allows your software to communicate with our AI services.
API Key
A unique identifier and secret token used to authenticate requests to the Elvion AI API. It verifies your identity and authorization.
Authentication
The process of verifying the identity of a user or application. For Elvion AI, this is done via the API Key in the Authorization header.
Bearer Token
A security token format used in HTTP authentication. It signifies that the bearer of the token (whoever has it) is authorized to access the resource.
Endpoint
A specific URL that represents a particular function or resource within an API (e.g., /api/developer/seraphina/chat).
Environment Variable
A dynamic named value that can affect the way running processes will behave on a computer. Used to securely store sensitive information like internal API keys on servers.
HTTP Status Code
A three-digit number returned by a server indicating the status of an HTTP request (e.g., 200 OK, 404 Not Found, 500 Internal Server Error).
Payload
The data sent in the body of an HTTP request or response. For Elvion AI, this is typically a JSON object.
Prompt Engineering
The art and science of crafting effective inputs (prompts) for AI models to achieve desired outputs.
Rate Limiting
A control measure that restricts the number of API requests a user or application can make within a given timeframe to prevent abuse and ensure stability.
Server-Sent Events (SSE)
A standard that enables a web page to receive automatic updates from a server over an HTTP connection, used for streaming responses.
Token
In the context of AI language models, a "token" is a piece of a word, character, or subword unit used for processing text. API usage is often measured in tokens.
